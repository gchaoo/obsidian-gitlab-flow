# AGENTS.md

本文件用于约束 Codex 在本插件目录内处理“3 个按钮业务规则”时的行为。

规则优先级：

- 用户明确要求优先
- 本文件中的按钮业务规则优先于 README 叙述性说明
- 若文档与代码不一致，先以当前代码行为为准，再决定是否同步更新 README / CHANGELOG / 测试

通用要求：

- 修改任一按钮行为时，先检查 `main.js`、对应工具函数文件和对应测试
- 涉及按钮业务规则变更时，默认同步更新 `README.md`
- 若行为变更会影响用户使用方式或结果格式，补充更新 `CHANGELOG.md`
- 不要擅自扩大功能范围，只按当前按钮所属业务链路做最小必要修改

## 按钮业务规则

### `同步会议纪要到 GitLab`

- 仅允许 `相关标签` 包含精确值 `会议纪要` 的 .md
- frontmatter 中的 `会议主题` 为空时，先回写为当前文件名
- 目标链接从 frontmatter 读取，字段名由设置项 `评论目标字段名` 决定，默认读取 `相关链接`
- 目标必须是 GitLab Issue 评论链接，必须包含 `#note_xxx`
- 该按钮只更新已有评论，不新建评论，不改 Issue 正文
- 同步前会移除指定 frontmatter 字段；当前默认忽略 `实时记录`
- 文档中的本地图片需要先上传到目标 GitLab 项目，再将正文中的图片链接替换为 GitLab 上传地址
- 远程图片保持原样
- 成功后不回写 Obsidian 文档内容

### `发布任务到 GitLab`

- 仅允许 `相关标签` 包含精确值 `PLM任务` 的 `.md` 文件发布
- `相关链接` 非空时更新已有 Issue
- `相关链接` 为空时，按 frontmatter `相关软件` 命中设置项 `软件项目映射` 后新建 Issue
- `相关软件` 为空、未命中映射或项目地址不合法时，直接报错终止
- `开始日期`、`结束日期` 必填，格式必须为 `YYYY-MM-DD`
- 发布前需要规范化文件名：
  - 移除旧式 `YYYY-MM-DD` 开头
  - 若文件名结尾缺少 `_yyyymmdd`，按 `开始日期` 补齐
- `任务名称` 为空时，回填为当前文件名，但要忽略开头的“数字-”前缀
- Issue 标题使用 `【相关合同】【相关软件】文章名称`
  - 保留文件名已有的 `_yyyymmdd` 后缀
  - 忽略文件名开头的“数字-”前缀
- GitLab 指派人取自最后一个“任务安排”表中的 `执行人` 列，并按表格数据行去重
- 发布成功后，将任务安排表中的执行人回写到 frontmatter `执行人`
  - 格式为 `[[张三]]` 这种 YAML 数组
- 发布成功后，将 frontmatter `开始日期`、`结束日期` 同步到 GitLab Issue 日期组件

发布时会更新文档中最后一个“任务安排”表：

- 若表中不存在 `PLM任务名称` 列，则在第一列自动补上该列
- 若表中已存在 `PLM任务名称` 列，则直接复用该列
- 每一行都要生成并更新 `PLM任务名称`
- `PLM任务名称` 规则为 `【合同名称】【软件名称】任务名称-任务类型_yyyymmdd`
- 生成 `PLM任务名称` 时，要去掉任务名称尾部已有的 `_yyyymmdd`
- 每一行都要更新 `时间范围`，格式为 `开始日期～结束日期`
- 保留其他列原值
- 若任一数据行缺少 `任务类型`，直接报错阻止发布

### `整理线上记录`

- 只处理当前打开的 `.md` 文件
- 只从正文 `## 千问记录` 标题下读取第一个可用链接
- 不再依赖 frontmatter `实时记录`
- 仅支持千问实时记录链接
- 若 `## 千问记录` 区块不存在、区块下没有可用链接，或链接不受支持，直接报错
- 执行后从页面中提取全文概要、章节速览和待办事项，整理成固定区块追加到正文末尾，不修改原文
- 重复执行时替换上一次生成结果，不重复追加
