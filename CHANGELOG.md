# Changelog

## 0.2.7

- 发布任务时的文件名规则从 `YYYY-MM-DD` 开头改为结尾 `_yyyymmdd`，缺少后缀时会按 `开始日期` 自动补齐
- 若文件名仍带旧式 `YYYY-MM-DD` 开头，发布时会自动移除旧前缀
- Issue 标题、PLM任务名称和 `任务名称` 回填值改为保留文件名中的 `_yyyymmdd` 后缀
- `任务名称` 回填值与 Issue 标题会忽略文件名开头的“数字-”前缀
- Issue 标题改为直接使用处理后的文章名称，不再重复追加一次日期后缀
- `PLM任务名称` 调整为按 `【合同名称】【软件名称】任务名称-任务类型_yyyymmdd` 逐行生成，生成时会去掉任务名称尾部已有的 `_yyyymmdd`
- 发布任务更新最后一个“任务安排”表时，若缺少 `PLM任务名称` 列，会自动在第一列补齐后再回填，不再直接报错
- `同步会议纪要到 GitLab` 改为仅允许 `相关标签` 包含精确值 `会议纪要` 的 Markdown 文档执行
- `同步会议纪要到 GitLab` 在 `会议主题` 为空时，会自动回填为当前文件名

## 0.2.6

- `整理线上记录` 改为读取正文 `## 千问记录` 区块下的千问链接，不再依赖 frontmatter `实时记录`
- 移除已不再使用的 `线上记录字段名` 设置项

## 0.2.5

- 发布任务成功后，会将任务安排表中的执行人回写到 frontmatter `执行人`，格式为带 `[[ ]]` 的多人数组
- 发布任务后会将 frontmatter `开始日期`、`结束日期` 同步到 GitLab Issue 的日期组件

## 0.2.4

- 发布任务时恢复保留 `PLM任务名称` 列，并按每一行生成 `【合同名称】【软件名称】任务名称_任务类型_yyyymmdd`
- 发布任务时若模板缺少 `PLM任务名称` 列，或某行缺少 `任务类型`，会直接报错阻止发布

## 0.2.3

- 移除“发布 GitLab 地址”和“发布目标项目名”设置项，任务新建不再依赖固定默认项目
- 新增“软件项目映射”设置项，改为键值对逐行编辑，支持新增一行、删除当前行、修改当前行，并按 `相关软件` 将任务发布到对应 GitLab 项目
- `相关链接` 为空时，若 `相关软件` 缺失、未命中映射或项目地址格式不合法，会直接报错提示

## 0.2.2

- 发布任务前会检查文章名，若缺少 `YYYY-MM-DD` 前缀则按 `开始日期 + 空格 + 原文件名` 自动重命名文档
- 发布任务时若 `任务名称` 为空，会自动回填为当前文档文件名
- GitLab Issue 标题改为基于文章名称生成，并自动忽略文章名开头的日期前缀

## 0.2.1

- 调整“发布任务到 GitLab”规则，仅允许带有 `PLM任务` 标签的 Markdown 文档发布
- 支持在 `任务名称` 为空时回退到文档文件名
- 发布时根据 `相关链接` 是否为空决定更新已有 Issue 或新建 Issue
- 发布时从最后一个“任务安排”表的 `执行人` 列提取并同步 GitLab 指派人
- 发布时移除最后一个“任务安排”表中的 `PLM任务名称` 列，并将每一行 `时间范围` 更新为 `开始日期～结束日期`

## 0.2.0

- 重命名插件为 `obsidian-gitlab-flow`
- 新增 `整理线上记录` 命令和文件菜单入口
- 支持从千问实时记录页面提取摘要、章节速览和待办事项
- 新增 `线上记录字段名` 设置项
- 统一现有命令和设置项文案
- 补充线上记录与设置布局测试
